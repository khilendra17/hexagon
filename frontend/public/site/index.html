THE PROBLEM WITH TRADITIONAL IV MONITORING

🩸

Blood Backflow Goes Undetected

Without automated sensors, backflow can occur unnoticed for minutes, risking contamination and severe patient harm.

💧

Manual IV Rate Checks Are Infrequent

Nurses check drip rates every 1–4 hours. A kink or mis-setting can under/over-dose a patient for hours without detection.

📡

No Correlation Between Drug & Response

Clinicians have no automated way to see when a drug started working, or if it's producing the expected physiological response at all.

🏥

Remote Monitoring Is Primitive

ICU staff cannot observe patients in rural or home-care settings without expensive, fragmented infrastructure that doesn't talk to each other.

HOW IT WORKS

ONE PLATFORM. COMPLETE CONTROL.

01

SENSE

ESP32 microcontroller reads IR sensor (drip rate), MAX30102 (SpO₂ + HR), and blood backflow detector in real time.

02

CONTROL

Solenoid valve auto-responds: increases, decreases, or halts flow based on real-time sensor thresholds per patient.

03

STREAM

Data pushed via MQTT/HTTP to Node.js backend + Socket.io for sub-200ms dashboard updates across all clients.

04

ANALYZE

AI layer correlates IV events with vitals, builds Drug Impact Curves, flags anomalies, and computes effectiveness scores.

05

ALERT

Multi-channel alerts: buzzer (device), push (Blynk mobile), and dashboard red-flag UI — simultaneously, within 200ms.

CORE CAPABILITIES

Built for the Critical Moment

IoT AUTOMATED

Intelligent Drip Rate Control

ESP32-driven IR sensor counts drop frequency and calculates real-time flow rate. The solenoid valve automatically corrects deviations, with configurable thresholds per patient profile. No nurse intervention required for minor adjustments.

PATIENT SAFETY

Blood Backflow Detection & Prevention

A dedicated backflow sensor continuously monitors pressure differential in the IV line. Upon detection, the solenoid valve closes instantly and a multi-channel alert fires — dashboard, mobile push, and audible buzzer — within under 200 milliseconds.

MAX30102 SENSOR

Continuous Vital Monitoring (SpO₂ + HR)

MAX30102 pulse oximeter and heart rate sensor streams photoplethysmography data. SpO₂ and heart rate are tracked at 1-second resolution, with configurable alert thresholds for both bradycardia and tachycardia events.

ESP32-CAM

Live Video Patient Surveillance

ESP32-CAM streams live patient video directly to the web dashboard and Blynk mobile app. The stream is overlaid with real-time vitals, creating a unified surveillance view — no separate camera system required.

REMOTE ACCESS

Remote Monitoring via Blynk + Web

Doctors and nurses can view live vitals, control IV flow rate, and acknowledge alerts from anywhere via the Blynk mobile dashboard or the web interface. Role-based access controls separate caregiver and administrator views.

< 200ms LATENCY

Multi-Channel Alert System

Simultaneous alerts across: onboard piezo buzzer (room-level), Blynk push notification (mobile), and dashboard red-flag UI. Alerts include event type, severity, timestamp, and patient ID — no ambiguity, no delay.

CORE INNOVATION · AI-POWERED DRUG INTELLIGENCE

DRUG IMPACT CURVE™

"See when the medicine starts working. Know when it isn't."

Every IV infusion event is automatically timestamped and correlated with the patient's continuous vital stream. The system computes response delay, effectiveness index, and trend deviation — then renders it as an interactive, time-series graph.

  • → Detects response latency: how many minutes until vitals respond to the drug
  • → Flags ineffective infusions when expected SpO₂/HR change doesn't materialize
  • → Generates a printable Treatment Response Report per patient per infusion
  • → Trained on timestamped IV event logs + physiological data using Python (scikit-learn / TensorFlow)

The correlation is anchored by a POST /api/events call — fired the moment an IV infusion begins. The system captures a 5-minute baseline of vitals before the event, then continuously tracks the delta. Response delay is the time from infusion start to the first statistically significant change in SpO₂ or heart rate. No machine learning — pure time-series rule logic, fully auditable.

AI MODULE · PYTHON + OPENCV

AI CAMERA INTELLIGENCE

"The system's eyes. Detecting what human eyes miss."

A dedicated Python FastAPI server receives live frames from the ESP32-CAM or an external IP camera. OpenCV processes each frame server-side — never on the ESP32 — and returns a structured detection result to the Node.js backend in under 300ms. No cloud dependency. No GPU required at baseline.

OPENCV · HSV THRESHOLD

Empty IV Bottle Detection

Fluid level is estimated by analyzing the transparent region of the IV bag. When the detected fluid zone drops below 15% of the bag area, an EMPTY alert is triggered and the solenoid valve prepares to close.

DETECTION LATENCY: < 300ms
OPENCV · COLOR DETECTION

Blood Backflow Detection

The IV tubing region is isolated in each frame. A hue-saturation filter detects the presence of red/dark-red color channels indicative of blood entering the line. Upon detection, the solenoid valve closes instantly and a CRITICAL alert fires across all channels.

TRIGGERS: SOLENOID CLOSE + ALERT
OPENCV · CONTOUR ANALYSIS

Air Bubble Detection

Circular contour detection identifies air pockets in the IV tubing. Bubbles above a minimum threshold size trigger a WARNING alert. This is an early indicator of an empty bag or improper line priming.

SEVERITY: WARNING

PHASE 8 · CLINICAL LANGUAGE OUTPUT

AI INSIGHT GENERATION

"Raw numbers become clinical decisions."

After computing the Drug Impact Curve and camera detections, the system generates plain-language clinical summaries. These are produced by rule-based logic — no black-box ML, no hallucinations — making them auditable and explainable to medical staff.

HOW IT WORKS — RULE LOGIC
IF response_delay < 3min

→ "Drug response was rapid. Effective."

IF response_delay > 10min

→ "Delayed response detected. Dosage review recommended."

IF spo2_delta < 1%

→ "No significant improvement in SpO₂ observed."

IF hr_delta within expected_range

→ "Heart rate response within normal parameters."
